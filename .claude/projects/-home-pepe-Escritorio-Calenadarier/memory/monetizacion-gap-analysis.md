# Monetization Gap Analysis – Calenadarier

## Summary
This document outlines the gaps between the current Calendario‑Web MVP and the freemium + SaaS model described in `MONETIZACION.md`. It details missing or incomplete features for each tier (Free, Pro, Team, Enterprise) and provides concrete actions to close those gaps.

## Gaps by Tier

### Free tier (ads, limits)
- No enforcement of user‑plan limits (max 10 people, 2‑month availability, max 3 calendars created, max 3 simultaneous calendars joined).
- No ad‑display component for free users.
- Missing subscription/plans table to flag a user as “free”.

### Pro tier
- Missing export endpoints (Excel, CSV, PDF).
- No audit‑log table or mutation hooks.
- Limited statistics (basic summary only).
- No dark‑mode/theme toggle.
- No mechanism to hide ads based on plan.

### Team tier
- No role‑based permissions (admin/editor/viewer) on people.
- No organization/workspace model – users can only create a personal calendar.
- Missing notification service (email/SMS/WhatsApp reminders).
- No external calendar sync (Google/Outlook).
- No import from Excel/CSV.
- No template gallery for sector‑specific calendars.
- No public API documentation or rate‑limiting.

### Enterprise tier
- No hierarchical org structure (company → dept → team → employee).
- No manager‑view‑all‑availability privacy controls.
- No SSO (SAML/OIDC) integration.
- No on‑premise/VPC deployment guide.
- No SLA, monitoring, or alerting setup.
- No custom branding (logo/colors per org).
- Extended audit‑login (login/logout, data export, plan changes) missing.
- No built‑in training/tour mode.

## Immediate Actions (MVP → Monetizable)

1. **Add subscription/plan field** to `users` (or new `subscriptions` table) and middleware (`requirePlan`) that checks limits before mutating data (create person, plan, availability >2 mo, etc.).
2. **Implement ad banner** component shown only when `user.plan === 'free'`.
3. **Create export endpoints** (`/api/calendars/[slug]/export?type=xlsx|csv|pdf`) using `xlsx` / `json2csv` / `jsPDF`.
4. **Add audit_log table** and wrap all mutation handlers (availability, people, plans, events) to insert change records.
5. **Upgrade statistics endpoint** (`/api/calendars/[slug]/stats`) with per‑person occupancy, trends; integrate a chart library (Recharts/Chart.js) in `ResumenView`.
6. **Add theme toggle** (via `next-themes` or CSS vars) and persist selection.
7. **Add role column** to `people` and UI in `SetupView` for admins to set role; enforce role‑based middleware.
8. **Design organization schema** (`organizations`, `user_organizations`) and migrate calendar ownership to org‑id; allow multiple calendars per org.
9. **Implement notification service** (Resend for email, optional Twilio/WhatsApp) and cron/webhook to send shift reminders.
10. **Add import endpoint** (`/api/calendars/[slug]/import`) accepting `.xlsx`/`.csv` and parsing into availability/events.
11. **Create calendar templates** table and endpoint to apply a template when creating a new calendar.
12. **Expose OpenAPI/Swagger docs** for existing API; add API‑key auth and rate limiting for external consumers.
13. **Prepare SSO groundwork** (add `next-auth` or similar) for future Enterprise enablement.
14. **Draft Docker‑Compose & deployment guide** for on‑premise/VPC installs.
15. **Add basic monitoring** (uptime ping, error logging) and create a simple status page (future SLA base).
16. **Implement custom branding** – allow org admin to upload logo & set primary color; use CSS variables to theme the UI.
17. **Extend audit log** to capture sign‑in, sign‑out, plan changes, data exports.
18. **Add onboarding/tour** (using Shepherd.js or similar) for Enterprise clients.
